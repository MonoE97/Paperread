Paperread V2 自包含 Skill Repo 行动方案

目标是把当前仓库从“repo-local Python 项目 + 薄 skill wrapper”迁移为真正的 skill repo：复制仓库中的 skill/ 目录到 Codex 或 Claude 的 skill 位置后，即可独立安装、运行、测试和使用。这里的“自包含”指文件 artifact 自包含；不同工作流仍需要 uv、Python、Zotero MCP、Node/CDP 等外部运行前提。

日期：2026-06-30 状态：可执行前 review 版 范围：计划与迁移方案，不包含实际迁移执行 安装名：paperread

1. 结论

建议执行方案：采用“Skill-as-package”结构。仓库内仍保留源目录名 skill/，但用户安装时复制到目标目录 paperread/，使安装目录名与 SKILL.md 中的 skill name 一致。

这个方案最直接地满足三个约束：所有必备项目文件都在 skill/ 内；继续使用当前稳定的 uv + Python CLI；同时兼容 Codex 和 Claude 的 skill folder 使用习惯。

不建议把 CLI 发布到 PyPI 或只保留轻量 skill wrapper。那会引入外部分发、版本漂移和离线不可用问题，和“即插即用复制一个目录”的目标冲突。

2. 已确认决策

问题 决策 理由
分发模型 完全自包含 复制 skill/ 后不依赖 repo root 文件；外部 runtime 作为前提单独声明。
环境管理 只支持 uv 符合项目既有纪律，也避免 pip/conda/global install 的不确定性。
功能范围 Zotero title workflow 和 local PDF path workflow 都保留 两条链路已经被当前 CLI 和测试覆盖，删掉任一条都会让 V2 功能缩水。
根目录角色 发布壳和维护文档 根目录不再是运行项目；运行、测试、模板、脚本和 lockfile 都进入 skill/。
目录命名 repo 内叫 skill/；安装后叫 paperread/ 同时满足你的仓库约束和 skill folder name 应匹配 skill name 的规范。

3. 规范依据

来自 superpowers:brainstorming

  • 先完成设计与计划，不在批准前执行迁移。
  • 现有代码库必须先读现状，再设计结构。
  • 设计文档要自查未决标记、矛盾、范围和歧义。
  • 用户 review 通过后，再进入实施计划或执行阶段。

来自 skill-creator

  • SKILL.md 必须有 name 和 description frontmatter。
  • SKILL.md 要精简，细节放到直接链接的 references/。
  • 确定性、易出错、重复执行的逻辑应放进脚本或 CLI。
  • skill 内不放 README、安装指南、changelog 这类辅助文档。
  • skill 内置自包含验证脚本；系统 quick_validate.py 只作为维护者交叉检查。

Claude 侧的兼容假设已经按官方文档校准：Claude Code 个人 skill 通常位于 ~/.claude/skills/<skill-name>/SKILL.md，因此安装后的目录应命名为 paperread。

5. 安装与使用模型

Codex 安装

install_dir="${CODEX_HOME:-$HOME/.codex}/skills/paperread"
test ! -e "$install_dir" || { echo "target exists: $install_dir"; exit 1; }
mkdir -p "$(dirname "$install_dir")"
cp -R /path/to/Paperread/skill "$install_dir"
cd "$install_dir"
uv sync --locked
uv run paperread --help

Claude Code 安装

install_dir="$HOME/.claude/skills/paperread"
test ! -e "$install_dir" || { echo "target exists: $install_dir"; exit 1; }
mkdir -p "$(dirname "$install_dir")"
cp -R /path/to/Paperread/skill "$install_dir"
cd "$install_dir"
uv sync --locked
uv run paperread --help

所有工作流命令都从安装后的 skill root 执行。V2 文档中不再出现“从 repo root 运行 uv run paperread ...”这种 V1 假设。

安装边界：目标 paperread/ 必须不存在或为空。若目标已存在，停止并让用户确认替换策略，避免复制出 paperread/skill/SKILL.md 这种无法被 Codex/Claude 正确发现的嵌套结构。

外部运行前提矩阵

能力 外部前提 说明
CLI 安装与运行 uv 和 Python >=3.13 依赖从 skill/uv.lock 安装到本地 skill 环境。
本地 PDF workflow 不需要 Zotero 只使用 bundled Python package 和依赖。
Zotero title workflow Zotero Desktop 与 Zotero MCP 写入仍然只能通过 MCP write_note，且必须有用户明确写入意图。
Secondary web capture Node.js 与 CDP helper 这是可选 cross-check 路径；不可用时应降级为 secondary_context_unavailable。
Skill 验证 skill/scripts/validate-skill.py 便携验证不依赖维护者机器上的个人系统 validator 路径。

6. Skill 设计边界

模块 职责 关键约束
SKILL.md 触发、路由、核心安全边界 保持精简；只保留 agent 必须立即知道的规则。
references/zotero-workflow.md Zotero title/title-fragment 工作流 写入只能通过 Zotero MCP write_note，且需要用户明确写入意图。
references/pdf-path-workflow.md 本地 PDF path 工作流 只写本地输出；禁止写 Zotero、禁止生成 write-payload.json。
references/summary-schema.md summary/review JSON 最小字段约束 正文中文优先，证据 locator 必须引用 canonical evidence。
src/paperread/ PDF 抽取、note 渲染、gate、payload、验证等确定性逻辑 复杂可靠性逻辑不写进 prompt，由 CLI 执行。
tests/ 行为验证和文档契约验证 测试必须能在复制后的 paperread/ 目录内独立运行。
scripts/validate-skill.py 自包含 skill 结构验证 验证 SKILL.md frontmatter、必备目录文件、无 in-skill README/安装指南/changelog。

7. 迁移执行顺序

阶段 0
开工前检查。
  • 确认当前分支、ahead/behind、工作区是否干净。
  • 创建本地工作分支，例如 codex/paperread-v2-skill-repo。
  • 不 push、不发布、不改全局环境。
阶段 1
结构迁移。
  • 移动 src/、templates/、tests/、pyproject.toml、uv.lock 到 skill/。
  • 更新 skill/pyproject.toml，删除 readme = "README.md"，不在 skill/ 内新增 README。
  • 从 skill/ 运行 uv lock，再用 uv sync --locked 验证 lockfile。
  • 检查 note.py 的 template path 和 cli.py 的 run dir resolver 是否仍指向 skill root。
阶段 2
Skill 元数据与 progressive disclosure。
  • 更新 skill/SKILL.md：描述 V2 自包含模型、skill-root 命令、两个工作流路由。
  • 新增或生成 skill/agents/openai.yaml。
  • 新增 skill/scripts/validate-skill.py，作为复制出去后仍可运行的结构验证脚本。
  • 更新 references/，移除 repo-root 假设。
阶段 3
测试与文档重写。
  • 让 skill/tests/ 不依赖根目录 README。
  • 更新 root README.md、README.zh-CN.md、AGENTS.md。
  • 增加维护者 root-doc review gate：python docs/superpowers/scripts/validate-root-docs.py 检查中英文 README、AGENTS 与 skill/SKILL.md 对安装路径、uv sync --locked、skill-root 命令、Zotero 写入红线、本地 PDF local-only 边界表述一致。
  • 调整 .gitignore，继续忽略本地运行产物、PDF、分析目录、缓存。
阶段 4
验证和复盘。
  • 从 skill/ 运行完整验证命令。
  • 复制 skill/ 到 repo 外临时 paperread/ 目录做强制安装形态验证；这不是可选 forward-test。
  • 运行维护者 root-doc review gate：python docs/superpowers/scripts/validate-root-docs.py。
  • 复查 diff，确认没有丢失 Zotero 写入红线、证据 locator 边界和 PDF local-only 边界。

8. 必跑验证门禁

迁移完成后，以下命令必须从 skill/ 内运行：

cd skill
uv sync --locked
uv run pytest
uv run paperread --help
uv run paperread extract-pdf tests/fixtures/minimal.pdf --output /tmp/paperread-extract.json
uv run python scripts/validate-skill.py .

以下复制目录门禁是证明 skill/ 自包含的核心证据，必须在 repo 外临时目录执行：

tmp_dir="$(mktemp -d)"
cp -R skill "$tmp_dir/paperread"
cd "$tmp_dir/paperread"
uv sync --locked
uv run pytest
uv run paperread --help
uv run paperread extract-pdf tests/fixtures/minimal.pdf --output /tmp/paperread-extract.json
uv run python scripts/validate-skill.py .

额外的结构验证：

  • git status --short --branch --untracked-files=all
  • git diff --check
  • find skill -maxdepth 3 -type f | sort
  • 确认根目录不再有 runtime src/templates/tests/pyproject.toml/uv.lock
  • 确认 root README、README.zh-CN、AGENTS 与 skill 文档的安装和安全边界一致

9. Forward Testing 计划

确定性测试和强制复制目录门禁都通过后，再做 agent 使用形态验证。Forward testing 用来检验提示与工作流可用性，不替代复制目录硬门禁。

测试 做法 禁止事项
Codex local PDF 在已通过复制目录门禁的临时 paperread/ 中，用 tests/fixtures/minimal.pdf 跑 dry-run local output。 不能写 Zotero。
Zotero dry-run 用 Zotero title 触发准备链路，不写入，只检查 tool discovery、run dir 和 gate 路径。 没有用户明确批准时不能调用真实写入。
Claude filesystem shape 复制到临时 ~/.claude/skills/paperread 形态，运行 uv sync --locked、paperread --help 和 scripts/validate-skill.py。 不能依赖 repo root 文件。

10. 风险与处理

风险 处理方式
测试仍依赖根目录文档，复制后失效。 把 skill runtime 测试改成只检查 skill/ 内文件；root README 同步进入维护者 root-doc review gate。
SKILL.md 变成长 README。 只保留路由、命令入口和红线；详细步骤进入 references/。
pyproject.toml 引用不存在的 README。 删除 readme = "README.md" 字段；不通过在 skill/ 内新增 README 来绕过。
验证依赖维护者机器上的个人系统 validator 路径。 把便携验证做成 skill/scripts/validate-skill.py；系统 quick_validate.py 只作为可选维护者交叉检查。
目标安装目录已存在，复制后生成嵌套结构。 安装命令必须先断言目标不存在；替换已有安装需要用户确认。
“自包含”被误解为不需要任何外部 runtime。 在计划和 README 中保留外部前提矩阵：uv、Python、Zotero MCP、Node/CDP 按工作流分别要求。
Zotero 写入边界在文档重写中丢失。 在 SKILL.md、Zotero reference、AGENTS、测试里重复保留 MCP-only write_note 红线。

11. 非目标

  • 不发布 PyPI、npm、marketplace 或 hosted service。
  • 不安装全局依赖，不修改系统 Python、conda base 或 shell 全局配置。
  • 不改变 note schema 或渲染布局，除非路径迁移迫使必要调整。
  • 不改变 Zotero 写入语义：仍然只能在用户明确要求后通过 MCP write_note 创建新子笔记。
  • 不移除 local PDF workflow。
  • 不在 skill/ 内创建 README、安装指南或 changelog。

12. 最终审核结论

当前判断：方案可以执行。剩余门禁不是设计问题，而是执行前确认：你确认后，再创建实施分支并开始迁移；在确认前不搬目录、不重写代码结构。
